## 1. 证据等级与评分区间

- **评审模式：** `design-craft / critique`
- **证据等级：L0 static**
  - 只有产品背景和对界面结构的文字描述。
  - 虽然用户、任务和风格目标明确，但没有截图或其他可检查的视觉产物，因此不能提升到 L1 contextual。
- **暂定 Product UI taste score band：60–74 / 100**
  - **成熟度：Functional but ordinary**
  - 从概念结构判断，更可能位于该区间的上半段，但现有证据不足以给出可信的单点分数。
- **评分含义：** 这是信息架构与产品适配度评分，不是对实际视觉执行质量的评分。

加分点：

- KPI、趋势和账户表覆盖了基本监控信息。
- 目标用户、决策时限和企业控制台风格已经定义。

主要扣分：

- 页面组织方式服务于“看全局数据”，而不是“在下一小时决定先处理什么”。
- 扁平 KPI 层级、装饰性图表和泛化建议构成典型的 dashboard card soup。
- 无法确认字体、间距、对齐、状态、无障碍和 token 执行质量。

## 2. Design read

> 这应该是一个为内部电商运营人员设计的、克制精确且数据密集的企业级决策台，优化目标是在三秒内识别最高影响的账户或活动，并明确下一步动作；但当前概念更像一张概览报表，而不是 operational decision surface。

核心任务不是“展示十二个指标”，而是完成这条决策链：

**发现异常 → 判断影响 → 理解原因 → 确认负责人 → 立即采取动作。**

任何不支持这条链路的模块，都应降级、后移或删除。

## 3. 阻塞性层级与产品适配问题

### B1 — P1：十二张等权 KPI 卡片抹平了运营优先级

所有指标拥有相同尺寸和 surface weight，意味着收入风险、例行总量和诊断指标被表达成同等重要。

用户必须自行阅读并重建优先顺序，页面没有承担 attention control 的责任。

### B2 — P1：缺少从“异常”到“行动”的首屏工作对象

当前结构有指标和表格，但没有明确的：

- 待处理异常队列；
- 影响金额或机会规模；
- 紧急程度与触发原因；
- 负责人、时限和下一步动作。

这会把运营人员留在“知道发生了变化，但不知道先做什么”的状态。

### B3 — P1：装饰性面积图消耗了注意力，却不回答运营问题

面积图如果只是展示总体走势，就很难解释：

- 哪个账户或活动造成变化；
- 变化是否越过阈值；
- 是短期波动还是持续异常；
- 用户应该采取什么动作。

它目前属于 decoration disguised as analysis。

### B4 — P1：密集表格可能是 data dump，而不是 task-first table

“密集”本身不是问题；企业控制台可以高密度。

问题在于，如果列顺序跟随数据 schema，而不是运营决策顺序，用户仍需横向解码才能找到状态、风险、影响和动作。表格会成为信息仓库，而不是工作队列。

### B5 — P1：通用 tips 右栏与当前实体和异常脱节

泛化建议既不构成证据，也不构成动作，还会压缩主表格的有效宽度。

如果建议没有明确的实体、触发条件、业务影响、负责人和动作，它不应占据永久性一级区域。

## 4. 八个具体设计动作

### 1. 建立紧凑的 page command/context band

在页面顶部集中放置：

- 当前业务范围；
- 时间窗口；
- 数据新鲜度或最后同步时间；
- 账户、渠道、负责人等全局筛选；
- “Needs attention”等运营视图。

避免营销式 hero。这里是控制上下文，不是品牌展示区。

### 2. 将 KPI card soup 改为 `lead + support` 指标层级

不要保留十二张等权卡片：

- 选择一个经过业务确认的 lead operational state，例如收入风险、目标缺口或紧急异常数量；
- 将四到六个支持指标压缩为 summary strip；
- 将诊断性或低频指标放入二级详情或可展开区域；
- 每个重要数值必须回答“相对哪个周期、基准或阈值”。

Lead 模块应紧凑而明确，不应变成巨型 hero card。

### 3. 把 exception queue 提升为首屏主要工作对象

按“影响 × 紧急度 × 可信度”排序账户或活动，并至少显示：

- 实体名称；
- 异常类型和触发阈值；
- 预估收入风险或机会；
- 异常持续时间；
- 数据新鲜度；
- 当前负责人；
- 明确的下一步动作。

验收目标：用户能在三秒内指出“第一件该处理的事”和原因。

### 4. 把面积图改造成 diagnostic chart，否则删除

先给图表定义一个具体问题，例如：

> 哪些活动导致过去 24 小时的收入缺口扩大？

然后再决定图表形式，并补齐：

- 基准线和异常阈值；
- 时间范围；
- 关键事件或投放变更注释；
- 实体分解或筛选；
- 与 exception queue、表格选中行的联动；
- 清晰的轴、单位、图例和 tooltip。

如果不能回答明确问题，应将其移到二级分析页或直接删除。

### 5. 将账户表改为 task-first table

优先列顺序建议为：

**账户/活动 → 状态 → 风险原因 → 影响金额 → 趋势/新鲜度 → 负责人 → 下一步动作**

同时：

- 文本左对齐，数值右对齐并使用 tabular numerals；
- 筛选器紧邻其影响的数据；
- 次要 metadata 延后、分组或按需展开；
- 行级动作必须可发现、可键盘访问；
- 长名称、超大数字和横向溢出需要有明确降级策略。

### 6. 删除 generic tips，或改造成 contextual action rail

右栏只有满足以下内容模板时才应保留：

**实体 + 观测变化/阈值 + 业务影响 + 证据 + 建议动作 + 负责人**

例如，它可以显示当前选中异常的原因、处理步骤和 owner，而不是固定展示通用运营建议。

在没有足够上下文的默认状态下，宁可折叠或移除右栏，把宽度还给主工作区。

### 7. 用克制的 enterprise-dense 视觉系统建立 density gradient

- 使用 token-backed 的中性背景、分隔线、文字和语义状态色；
- 颜色只表达风险、成功、警告、选择等语义，不承担装饰；
- 常规内容优先使用平面 section、row anatomy 和细分隔线；
- elevation 只用于 overlay、selection 或真正的层级变化；
- 通过字号、字重、间距和数值对齐建立层级，而不是增加边框和彩色卡片；
- 高风险区域密度更高、对比更明确；支持性信息更安静。

### 8. 把 resilient states 纳入组件合同

在实现前定义并设计：

- loading、empty、error；
- stale、partial data、permission denied；
- disabled、selected、hover、focus-visible；
- 超长账户名、极端数值、无负责人；
- 窄视口和表格溢出；
- 非颜色依赖的风险表达；
- 键盘导航、tooltip 可达性和错误恢复路径。

这些不是后续 polish，而是运营工具可信度的一部分。

## 5. 已验证与未验证

### 仅由提示词确认的事实

- 产品是内部电商 Revenue Operations dashboard。
- 主要用户要决定下一小时优先处理的账户或活动。
- 当前概念包含十二张等权 KPI 卡片、装饰性面积图、密集账户表和通用 tips 右栏。
- 目标风格是克制、平静、token-backed 的企业控制台。
- 没有提供截图、DOM 或运行态证据。

### 基于结构作出的高置信度设计推断

- 等权 KPI 会削弱业务优先级表达。
- 装饰性图表不直接支持下一步决策。
- 泛化 tips 与当前实体和异常的关联不足。
- 当前页面更偏 overview，而不是 triage/action workflow。

这些是信息架构判断，不代表已经测得真实的操作耗时或错误率。

### 完全未验证

- 实际视觉层级、间距、对齐、字体和表面质量；
- 是否真正遵循设计 token；
- 色彩对比度及状态色语义；
- 表格列结构、排序、筛选和行级动作；
- 图表数据尺度、标签、tooltip 和响应式表现；
- hover、focus、selected、loading、empty、error 状态；
- 键盘与屏幕阅读器可用性；
- 长内容、真实数据和窄视口表现；
- 页面性能、数据延迟和交互反馈。

**本次没有进行浏览器验证、截图验证、DOM 检查或文件检查，也没有实施任何改动。**

## 6. 实现前的最小验证计划

1. **获取真实基线 artifact**  
   至少提供当前界面截图或可检查的 Figma/wireframe；若已有运行页面，再补充 DOM、token 和组件使用情况。否则评分必须继续保持 L0。

2. **复盘真实的下一小时决策**  
   用最近三到五个运营案例记录：触发信号、所需证据、判断、负责人、动作以及处理时限。

3. **确认数据与排序合同**  
   明确 KPI 定义、时间口径、对比基准、异常阈值、数据延迟，以及 exception queue 的排序规则。不要由视觉设计自行猜测 lead metric。

4. **用真实内容制作低保真 decision surface**  
   使用真实长度的账户名、活动名、金额、异常原因和缺失字段，而不是理想化 placeholder。

5. **执行三秒识别测试**  
   让至少三名目标用户回答：“现在第一件该处理什么、为什么、影响多大、下一步是什么？”记录识别时间和正确率。

6. **确定实现验收矩阵**  
   实现完成后至少检查主桌面与窄桌面视口，以及 loading、empty、error、stale、long-content、focus、keyboard、selected 和 chart tooltip 状态；在产生真实截图与 DOM/token 证据之前，不应宣称达到 L2 或更高证据等级。
