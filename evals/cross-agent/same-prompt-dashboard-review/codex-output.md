## 1. 证据等级与评分区间

- **使用模式**：`design-craft / critique`
- **证据等级：L0 static**
  - 只有产品描述和概念结构。
  - 虽然已知用户与核心任务，但没有截图，因此不能提升到要求“截图 + 产品上下文”的 L1。
- **暂定产品适配评分：60–74 / 100**
- **区间：Functional but ordinary（功能框架存在，但产品判断与任务层级粗糙）**
- **置信度：低**
- 评分仅覆盖概念层面的信息顺序和产品适配，不代表真实视觉完成度。
- 如果实际排版、字体、间距和 token 使用非常成熟，视觉分可能更高；但只要“12 个等权 KPI + 装饰图表”的 `card soup` 仍是主结构，整体不应高于 84。

**一句话诊断：** 当前概念更像“业务概览页”，而用户需要的是“下一小时异常处置台”。

## 2. Design read

**Reading this as：** 面向内部电商运营人员的 revenue-ops 决策台，采用 `enterprise dense + ops command center` 的克制气质，优化目标是在三秒内找到最需要关注的账户或活动，十秒内理解原因与影响，并立即执行或分派下一步动作。

它不应优化为“尽可能展示更多数字”，而应优化为：

1. **What needs attention?** 哪个账户或活动最紧急？
2. **Why now?** 触发原因、偏差幅度和数据新鲜度是什么？
3. **What is the impact?** 收入风险、增长机会或预算损耗有多大？
4. **What should I do?** 下一动作是什么，由谁负责？
5. **Where is the evidence?** 如何进入趋势、明细和诊断依据？

## 3. 五个阻塞性问题

### 1. 等权 KPI 网格抹平了运营优先级

12 张尺寸、位置和表面权重相同的卡片，会让例行总量、异常指标和真实收入风险争夺同等注意力。

问题不是 KPI 数量本身，而是没有形成 `lead / support / diagnostic` 层级。用户必须先阅读和解释页面，才能知道该做什么。

### 2. 缺少真正的异常队列与下一动作

用户任务是决定“接下来一小时处理谁”，但当前结构没有明确的 `lead risk object` 或 `exception queue`。

最高风险对象很可能被埋在密集表格中；这使页面能展示状态，却不能主动帮助排序、解释和处置。

### 3. 装饰性面积图没有承担诊断职责

面积图如果没有回答明确问题，例如“异常从何时开始”“偏离哪个基准”“哪个账户贡献了变化”，就只是高权重装饰。

它占据本应留给异常解释、影响判断或行动入口的首屏空间，同时可能制造“有分析能力”的错觉。

### 4. 密集表格尚未证明是 `task-first table`

“密集”对企业控制台不是缺点，**无任务顺序的密集才是问题**。

目前没有证据表明表格能优先扫描：

`identity → status/anomaly → impact → urgency → owner → next action`

也没有证据证明筛选、排序、行操作、数据新鲜度和诊断入口与用户决策相邻。

### 5. Generic tips 右栏占用了错误的空间

通用技巧与当前账户、异常原因和可执行动作没有上下文关系，属于低价值常驻模块。

它压缩了主表宽度，却没有减少用户判断成本。对这个产品，右栏只有在提供上下文诊断或行动支持时才成立。

## 4. 八个具体设计动作

### 1. 改成 `lead + support + exception queue + diagnostic + task-first table`

首屏顺序建议：

1. 当前范围、时间窗口和数据新鲜度；
2. 一个紧凑的首要风险对象；
3. 支撑型指标条；
4. 高优先级异常队列；
5. 与当前异常关联的诊断图；
6. 完整账户或活动表格。

这是一张 `decision surface`，不是平均铺开的组件目录。

### 2. 设置紧凑的 `lead risk object`

不要做营销式大 Hero，而是突出当前最需要处理的账户或活动：

- 对象名称；
- 异常原因；
- 收入风险或增长机会；
- 相对阈值或基准的偏差；
- 持续时间与最新更新时间；
- 当前负责人；
- 一个明确的 verb-object 动作，例如“查看预算异常”或“分派给账户负责人”。

### 3. 把 12 个 KPI 压缩为 4–6 个支持指标

按运营决策分组，而不是按数据源分组，例如：

- 收入与目标偏差；
- 投放效率；
- 预算或交付风险；
- 异常账户数量。

每个被强调的数字都应回答：

- 哪个时间范围？
- 相比什么？
- 阈值是什么？
- 数据更新于何时？

其余指标进入二级详情或可展开区域，不再各占一张等权卡片。

### 4. 增加可解释的异常队列

异常队列应位于表格之前，并按照明确的优先级规则排序，例如：

`预计影响 × 紧迫度 × 置信度`

每项至少显示：

- 账户或活动；
- 异常类型；
- 影响金额或机会；
- 触发原因；
- 持续时间；
- owner；
- 推荐下一动作。

优先级必须可解释，不能只显示不透明的“风险分”。

### 5. 将面积图改为选中异常的诊断图

图表应回答一个问题，而不是仅显示趋势。

例如：

- “该账户的收入从何时开始偏离基准？”
- “预算消耗与转化下降是否同时发生？”
- “这次总收入下滑由哪些活动贡献？”

使用明确时间范围、基线、阈值和必要标注；队列或表格选择变化时更新图表。如果没有值得回答的问题，宁可移除图表。

### 6. 重构为 `task-first table`

建议优先列序：

`账户/活动 → 状态与异常 → 收入风险/机会 → 偏差 → 持续时间 → owner → 下一动作`

实现约束：

- 文本左对齐，数值右对齐；
- 使用 tabular numerals；
- 筛选和排序紧邻表格；
- 首要标识和必要操作按实际滚动需求决定是否 sticky；
- 次要 metadata 分组或渐进披露；
- 行动作需要明确、可发现且可键盘访问。

### 7. 移除或重构右栏

两个合理方向：

- **首选：** 收回宽度给主表和异常队列；
- **备选：** 改为随当前选中异常变化的上下文行动栏，展示原因、证据、负责人、备注和动作。

不要保留与当前决策无关的常驻 generic tips。

### 8. 建立克制的 token-backed 状态层级

采用 `enterprise dense` 处理：

- 中性页面和表面 token；
- 细分隔线，最小 elevation；
- 颜色只表达语义状态，不装饰 KPI；
- critical、warning、positive、neutral 使用统一 token；
- 状态同时使用文字或图标，不能只依赖颜色；
- 为 loading、empty、error、stale、permission、long-label 和 narrow-viewport 明确定义布局；
- 所有交互元素保留可见的 `focus-visible`。

## 5. 已验证与未验证

### 仅由提示词确认

- 产品是内部电商 revenue-ops dashboard。
- 核心用户需要决定下一小时优先处理哪个账户或活动。
- 概念包含 12 个等权 KPI、装饰性面积图、密集表格和 generic tips 右栏。
- 风格权威要求克制、密集但平静、无营销 Hero、只使用 token-backed color。
- 当前信息架构与“快速分诊并采取动作”的核心任务存在结构性错位。

最后一项是基于已描述结构得出的设计推论，不是来自真实用户行为测试。

### 未验证

- 实际视觉层级、网格、间距、字体和表面质量；
- 是否真实使用设计 token，以及明暗主题是否一致；
- 色彩对比度和状态是否仅依赖颜色；
- KPI 的时间范围、基准、阈值、单位和数据新鲜度；
- 图表的比例尺、标签、图例、tooltip 和真实分析价值；
- 表格列序、筛选、排序、分页、横向滚动和行操作；
- hover、focus、keyboard、loading、empty、error、stale 和 permission 状态；
- 长名称、大数字、异常数据量和真实内容下的布局；
- 响应式行为、窄视口降级策略和性能；
- 用户能否在三秒内正确找到最高优先级对象。

**没有执行浏览器验证、截图验证、DOM 检查、响应式运行或文件修改。**

## 6. 实现前的最小验证计划

### Gate 1：补齐一份 L1 证据包

至少提供：

- 一张包含接近真实数据的完整桌面截图或高保真稿；
- 12 个 KPI 的业务含义、时间窗口和优先级；
- 表格列、筛选、排序和行动作清单；
- 图表计划回答的业务问题；
- 右栏内容与使用场景。

这一步用于判断当前问题究竟只是结构错误，还是还存在明显的视觉系统问题。

### Gate 2：定义“下一小时注意力”合同

与产品或运营确认：

- 最常见的 3–5 类异常；
- 每类异常的影响、紧迫度和排序依据；
- 什么信息足以采取动作；
- 哪些动作可以在 dashboard 内完成；
- 谁负责，以及如何显示数据新鲜度。

如果无法定义优先级规则，就不应先实现异常排名 UI。

### Gate 3：用一个低保真重排稿做任务测试

使用接近真实的数据，让 2–3 名目标运营人员完成三个任务：

1. 找到当前最高优先级账户或活动；
2. 说明为什么它最紧急；
3. 找到并描述下一动作。

最低验收：

- 无需逐张阅读所有 KPI；
- 首要对象与原因能在约 3–5 秒内被识别；
- 用户不需要进入表格横向解码才能决定第一步。

### Gate 4：在编码前冻结状态与视口合同

至少定义：

- 默认、loading、empty、error、stale、permission 状态；
- 长账户名、大金额、负值和大量行；
- 最小支持视口，以及右栏和表格在窄视口下的降级顺序；
- keyboard、focus-visible 和非颜色状态表达；
- 使用哪些现有 surface、text、border、status 和 focus tokens。

这四项通过后，才有足够证据进入实现；真实浏览器、截图、响应式和交互状态验证应作为实现后的独立验收门禁。
