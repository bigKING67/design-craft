## 2. Design read

> Reading this as: 面向内部电商运营人员的高密度 Revenue Operations 决策台，采用克制、安静的企业控制台语气，优化目标是在几秒内识别、解释并处理未来一小时内影响最大的账户或活动异常。

它首先应该是一个 **decision surface**，其次才是指标浏览器。

核心决策链应当是：

**哪里出问题 → 影响多大 → 为什么值得现在处理 → 谁负责 → 下一步做什么**

## 3. 阻断性层级与产品适配问题

### 1. P1 — 12 个等权 KPI 消除了注意力控制

题面明确说明 KPI 全部等大，这会把高风险异常、常规总量和诊断指标表达为同等重要。

用户需要先自行比较 12 个数字，再推断优先级；页面没有直接回答“现在先处理什么”。

### 2. P1 — 面积图占据视觉预算，但没有承担诊断任务

“装饰性”图表没有明确分析问题、基准线、异常阈值或与账户/活动的联动关系。

它不仅没有帮助决策，还会与真正需要注意的异常对象争夺首屏注意力。

### 3. P1 — 表格有密度，但概念中没有定义 task-first anatomy

高密度本身适合企业运营界面，问题不是“表格太密”，而是没有说明是否优先呈现：

**对象身份、状态、风险、业务影响、负责人、更新时间、下一动作。**

若列顺序来自数据模型而非决策流程，表格只是数据仓库的视觉投影。

### 4. P1 — Generic tips 右栏占用固定空间，却没有上下文价值

泛化建议通常不与当前异常、所选账户、影响金额或运营权限相关。

在持续使用的内部工具中，这类内容很快会形成 banner blindness；固定右栏还会压缩更重要的表格扫描宽度。

### 5. P1 — 概念尚未定义“需要关注”的产品判定合同

视觉层级无法替代业务优先级模型。当前描述没有明确：

- 影响金额或机会损失；
- 异常严重度与紧迫度；
- 判断置信度；
- 数据新鲜度；
- 负责人和处理状态；
- 推荐动作及恢复路径。

没有这组语义，任何“突出显示”都可能只是视觉猜测。

## 4. 八个具体设计动作

1. **建立 `lead risk object`**
   - 首屏突出一个最高优先级账户或活动，而不是突出一个抽象总量。
   - 同时显示影响、异常原因、持续时间、数据新鲜度、负责人和一个明确的 verb-object 动作。

2. **将 12 卡网格改为 `lead + support + action queue`**
   - 保留 1 个主风险对象。
   - 将 4–6 个必要 KPI 压缩为 supporting metric strip。
   - 其余诊断指标进入详情或按业务分组展示，不继续制造等权卡片。

3. **让每个强调数字回答“与什么相比”**
   - 为指标补充时间范围、目标、基线、阈值或同比/环比。
   - 将状态分为 lead、support、diagnostic tiers，而不是仅调整字号。
   - 正负变化必须基于业务语义，不能机械地把增长都标绿、下降都标红。

4. **在首屏建立 exception queue**
   - 展示最需要处理的 3–5 个账户或活动。
   - 每行至少包括严重度、影响、异常年龄、负责人、处理状态和下一动作。
   - 排序规则应由影响、紧迫度和置信度决定，并向用户解释排序依据。

5. **把账户表改造成 task-first table**
   - 推荐顺序：`Account/Campaign → Status → Risk/Impact → Cause signal → Owner → Updated → Next action`。
   - 数字右对齐并使用 tabular numerals；文本左对齐。
   - 次要元数据通过列配置、展开行或详情面板渐进披露。
   - 过滤器与其影响的数据相邻，而不是散落在页面其他区域。

6. **把面积图改成 diagnostic chart**
   - 先确定问题，再选图表，例如“实际收入何时偏离目标”“哪些账户贡献了异常”“异常是否仍在扩大”。
   - 加入目标线、阈值、异常标记和可读的时间范围。
   - 与 exception queue／表格选中项联动；删除没有分析意义的面积填充和装饰效果。

7. **删除 generic tips，或改为 contextual inspector**
   - 右栏可用于展示当前所选异常的证据、原因、历史动作和建议处理路径。
   - 如果没有足够上下文内容，则直接移除右栏，把宽度还给表格。
   - 在较窄笔记本视口下改为抽屉或行内详情，避免永久挤压主任务。

8. **建立克制的企业级 surface 与状态合同**
   - 使用低对比度背景、细分隔线和间距形成分组，减少独立卡片与无意义 elevation。
   - 中性色负责结构；红、橙、绿等 token 只表达已定义的状态。
   - 状态不能只依赖颜色，同时使用文本、图标或形状。
   - 明确定义 loading、empty、error、stale、permission、selected、hover、focus-visible、action success 和 recovery。

## 5. 已验证与未验证

### 仅由题面确认

- 产品是内部电商团队使用的 Revenue Operations dashboard。
- 主要用户需要决定未来一小时应关注哪个账户或活动。
- 当前概念包含 12 个等权 KPI、装饰性面积图、高密度账户表和 generic tips 右栏。
- 风格方向是克制、密集但平静的企业控制台，并要求使用 token-backed color。

### 基于上述事实作出的设计判断

- 等权 KPI 结构会削弱 operational priority。
- 装饰性图表不符合以行动为目标的 dashboard。
- generic tips 的固定空间价值低于异常队列或上下文详情。
- 推荐方向应从 overview dashboard 转为 decision surface。

这些是概念层面的设计推论，不是对真实渲染或用户行为的实测结论。

### 未验证，不能声称通过

- 实际视觉质量、间距、字体、对齐、颜色和对比度；
- 是否真的使用设计 token，以及 light/dark parity；
- KPI 定义、阈值、时间范围和数据新鲜度；
- 表格列顺序、排序、过滤、分页、粘性列和行操作；
- 图表比例尺、标签、图例、tooltip、异常表达和数据正确性；
- hover、focus、selected、disabled 和键盘操作；
- loading、empty、error、stale、permission 和 partial-data 状态；
- 长账户名、大数值、缺失值与真实数据量；
- 笔记本窄视口、移动端或其他响应式行为；
- 可访问性、性能及用户是否能更快完成任务。

本次没有编辑文件，也没有执行或声称任何浏览器验证。

## 6. 实现前的最小验证计划

1. **定义 attention contract**
   - 与运营人员确认“未来一小时需要关注”的判定规则。
   - 明确 impact、urgency、confidence、freshness、owner、status 和 next action。

2. **准备真实数据样本**
   - 至少覆盖正常、突发下降、预算异常、数据延迟、无人负责和多异常并发场景。
   - 包含长名称、极值、零值、缺失值以及足以验证密度的账户数量。

3. **制作低保真决策流**
   - 先验证 `lead + support + action queue + task-first table` 的信息顺序。
   - 不要在业务排序逻辑尚未确定前投入高保真视觉工作。

4. **用 3–5 名目标运营人员完成三个任务**
   - 3 秒内指出最高优先级对象；
   - 30 秒内解释它为何需要处理；
   - 60 秒内找到负责人并发起正确动作。
   - 同时记录误判、回看次数和依赖外部工具的步骤。

5. **验证主工作宽度**
   - 至少检查目标桌面宽度和较窄笔记本宽度。
   - 移动端只有在真实使用场景包含移动操作时才进入首轮范围，避免机械地把桌面布局纵向堆叠。

6. **在交付开发前冻结验收条件**
   - 首屏可识别最高风险；
   - 每个主指标具备比较上下文；
   - 表格一屏可扫描身份、状态、风险和动作；
   - 图表回答一个明确问题；
   - token、非颜色状态表达和关键状态矩阵已定义。

以上门禁通过后再进入实现；真实 DOM、截图、响应式、focus、状态和图表行为仍应作为实现后的浏览器验证门禁。
