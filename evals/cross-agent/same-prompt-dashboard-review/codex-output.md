## 1. 证据等级与评分带

- **方法**：`design-craft`，模式为只读 `critique`。
- **证据等级**：**L0 static**——只有概念描述和产品上下文；没有截图，因此不能提升为 L1 contextual。
- **暂定评分带**：**60–74 / 100，Functional but ordinary**。
- **置信度**：低。该评分只反映概念的信息架构和产品适配度，不是对实际渲染质量的测量。
- **主要加分项**：用户、决策时限和“克制的企业控制台”方向清楚。
- **主要扣分项**：明确存在 `card soup`、扁平优先级、装饰性图表和泛化提示。
- 实际排版、字体、间距、对齐、颜色、状态和响应式表现均未进入评分。

## 2. Design read

**Reading this as：面向内部电商运营人员的克制、精确、数据密集型 Revenue Operations 控制台，优化目标是在数秒内识别未来一小时最需要处理的账户或活动，并直接进入下一步动作；但当前结构更像“展示全部指标”，而不是“帮助完成分诊”。**

页面的主对象不应是 KPI 总览，也不应是趋势图，而应是：

> **当前最值得处理的异常实体、判断依据、潜在收入影响，以及可执行的下一步。**

## 3. 最重要的层级与产品适配问题

1. **[P1] 12 张等权 KPI 卡造成扁平优先级**
   - 所有指标占据相同面积和表面权重，用户仍需自行判断什么最重要。
   - 对“下一小时处理谁”的任务而言，这是典型的 `dashboard card soup`。
   - 总览指标占据第一注意力，却没有形成一个明确的 `lead operational state`。

2. **[P1] 决策队列被放在汇总和装饰内容之后**
   - 账户表才最接近用户的实际决策对象，但当前阅读顺序是 KPI → 图表 → 表格。
   - 这遵循了报告式信息顺序，而不是运营式的 `detect → assess → act` 顺序。
   - “密集”本身不是问题；缺少任务优先级的密集表才是问题。

3. **[P1] 面积图被定义为装饰性内容**
   - 图表没有被赋予明确的分析问题，例如“过去 60 分钟哪些账户恶化最快”。
   - 它占用高价值空间，却没有说明异常、影响或下一动作。
   - 对 `Operate` 型界面，装饰性数据可视化应删除，而不是继续润色。

4. **[P1] Generic tips 右栏与核心任务竞争**
   - 提示没有指定实体、变化、阈值、业务影响、负责人或动作。
   - 它既压缩了表格宽度，又容易制造“系统很智能但不可执行”的假象。
   - 泛化建议不应拥有常驻一级区域。

5. **[P1，规格阻塞] 优先级语义尚未定义**
   - 提示中未说明：判断窗口、数据新鲜度、异常阈值、潜在收入影响、优先级规则、负责人和可执行动作。
   - 这不能证明现有产品一定缺失这些能力，但在实施重构前必须形成明确合同。
   - 如果系统不能解释“为什么这个账户排第一”，视觉强调也不会可信。

## 4. 具体设计动作

1. **先建立可解释的分诊模型**
   - 明确优先级由哪些因素构成，例如紧急程度、收入影响、异常置信度、数据新鲜度和 SLA。
   - 每个高优先级实体必须能够回答：`Why now?`、`What is at risk?`、`What should I do?`
   - 不要只输出一个不透明的“风险分”。

2. **重组为 `lead + support + action queue`**
   - 第一视口依次放置：共享时间/渠道范围与数据新鲜度、一个核心运营状态、紧凑支持指标、优先行动队列。
   - 让队列成为面积和结构上的主导区域，而不只是依靠警示色吸引注意。
   - 历史诊断和解释内容移到队列之后。

3. **将 12 张等权 KPI 改成指标层级**
   - 分为 `lead`、`supporting`、`diagnostic` 三层。
   - 首屏只保留真正会改变分诊结果的指标；诊断指标进入展开区或实体详情。
   - 每个强调数字必须同时回答“相较于什么”：时间段、基线、阈值、目标或前一周期。
   - 使用 tabular figures 和一致的数值、单位、小数位及正负号规则。

4. **把账户表改成 `task-first exception queue`**
   - 建议优先列顺序：实体、严重度、收入影响、触发信号、最近变化/新鲜度、负责人、下一动作。
   - 默认排序应基于行动优先级，而不是数据库顺序或单一绝对值。
   - 数值右对齐、文本左对齐；次要元数据分组或渐进披露。
   - 关键行操作保持可见且键盘可达，不应只在 hover 时出现。

5. **让图表回答一个具名的运营问题**
   - 例如：“过去 60 分钟哪些账户的转化或消耗偏离最明显？”
   - 显示单位、时间范围、基线、异常阈值和直接标签；图表与受影响实体建立明确跳转关系。
   - 如果图表不能改变排序、判断或动作，应删除，而不是换一种图表样式。

6. **删除 generic tips，或改为上下文决策支持**
   - 默认把右栏宽度还给行动队列。
   - 选中一行后，可用 drawer 或 contextual panel 展示原因、证据、历史变化、建议动作和负责人。
   - 每条保留的 insight 必须包含：实体、观测变化/阈值、业务影响、下一动作或 owner。

7. **采用 calm-density，而不是“卡片化整齐”**
   - 使用一个主工作平面、克制分隔线和明确的 density gradient：决策区适度留白，支持数据紧凑，诊断细节更密。
   - 普通内容优先通过对齐、间距和背景层次分组，不为每个模块添加独立圆角卡片。
   - 颜色只来自语义 token；状态同时使用文字、图标或形状，不能仅依赖颜色。
   - 不引入 hero、渐变、玻璃效果或营销式大标题。

8. **把状态、响应式与可访问性写入组件合同**
   - 至少定义 loading、empty、no-action-needed、partial/stale、error/retry、permission、long-label 和 extreme-value 状态。
   - 窄屏不是简单堆叠：保留最高优先级实体和下一动作，次要 KPI 与元数据进入 disclosure。
   - 规定 `focus-visible`、键盘顺序、选中态、hover、禁用态和错误恢复；关键动作不能依赖 hover。

## 5. 已验证与未验证

**仅由提示明确确认：**

- 产品是内部电商 Revenue Operations dashboard。
- 主要用户需要决定未来一小时优先处理哪个账户或活动。
- 当前概念包含 12 张等权 KPI 卡、一张装饰性面积图、密集账户表和 generic tips 右栏。
- 设计 authority 要求克制、密集但平静、非营销化，并只使用 token-backed color。
- 本轮是只读评审，没有文件修改。

**基于以上事实作出的设计判断：**

- 当前页面结构与快速分诊任务不匹配。
- 等权卡片、装饰图表和泛化提示削弱了行动优先级。
- 账户/活动异常队列应成为主决策对象。

这些属于产品设计判断，不是浏览器运行态证明。

**未验证：**

- 实际视觉构图、像素间距、网格、字体、表面处理和数值对齐。
- 实现是否真正遵守颜色 token，以及对比度、主题和语义色是否合格。
- KPI 是否已经包含周期、基线、阈值、数据源和更新时间。
- 表格列、默认排序、过滤器、行操作、sticky 行为和横向溢出。
- 图表比例尺、标签、legend、tooltip、空数据和 resize 行为。
- hover、focus、keyboard、selected、disabled、loading、empty、error 和恢复状态。
- 桌面、窄屏或其他响应式布局。
- 真实数据下的长名称、极值、零值、负值、缺失值和权限差异。

**本轮未执行任何浏览器、DOM、截图或交互验证，也不宣称相关验证通过。**

## 6. 实施前的最小验证计划

1. **建立决策合同**
   - 与运营和数据 owner 确认时间窗口、优先级规则、阈值、收入影响口径、数据新鲜度、owner、SLA 和允许执行的动作。
   - 用三类场景校验：单一紧急异常、多个竞争异常、无可执行异常或数据过期。

2. **取得真实 baseline 证据**
   - 在后续独立验证中查看实际 dashboard，采集一个标准桌面视口和一个窄视口截图。
   - 同时检查相关 DOM、computed styles 和 token 来源。
   - 若没有可运行页面，则用含代表性真实数据的低保真原型代替，但证据仍停留在 L0/L1，不冒充浏览器证明。

3. **做最小运营任务测试**
   - 让 2–3 名目标运营人员回答：“未来一小时先处理谁、为什么、下一步是什么？”
   - 验收目标：在约 3 秒内找到首要实体；随后能够解释排序依据并找到明确动作。
   - 若不同人员持续得出不同优先级，先修正决策模型，而不是调整颜色强弱。

4. **锁定实现验收矩阵**
   - 覆盖桌面与窄屏、focus walk、keyboard、hover/selected、loading、empty、stale/partial、error/retry、permission 和长数据。
   - 对表格验证排序、过滤、行操作和溢出；对图表验证尺度、标签、tooltip、legend、空数据及容器 resize。
   - 只有真实浏览器、代表性数据和截图/状态证据完成后，才能把评审提升到 L2/L3 并给出更窄的实际 UI 评分。
