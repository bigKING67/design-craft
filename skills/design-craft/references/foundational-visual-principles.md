# Foundational visual principles

Use this as the compact visual reasoning layer behind taste review, polish, and
redesign decisions. It is intentionally small: project `DESIGN.md`, runtime
truth, and product context still outrank these generic principles.

## Core premise

Design organizes attention before decorating surfaces. A screen is successful
when users can see what matters, understand what belongs together, and choose
the next action without decoding the layout.

## Attention

- Decide what should be noticed first, second, and last.
- Increase emphasis only for priority, state, risk, action, or brand meaning.
- If everything is emphasized, nothing is important.

## Proximity

- Related objects sit close enough to be read as one group.
- Unrelated groups need enough space, divider, background shift, or section
  rhythm to be interpreted separately.
- Helper text, errors, labels, and actions should stay near the object they
  explain or affect.

## Alignment

- Every element should visually connect to a grid, edge, baseline, or rhythm.
- Drifted edges and inconsistent control heights make UI feel assembled rather
  than composed.
- Tables, metrics, and reports need stricter numeric and label alignment than
  editorial surfaces.

## Repetition

- Repeated meaning should use repeated treatment: type role, spacing, color,
  radius, surface, icon size, and state behavior.
- Meaningful differences should be visible; accidental differences should be
  removed.
- A component family is not just similar shape; it shares anatomy and states.

## Contrast

- Different importance, action strength, state, risk, or role must look
  different enough to scan quickly.
- Weak contrast feels timid; excessive contrast feels noisy.
- Prefer contrast in hierarchy, typography, density, and placement before
  adding decorative color or heavy elevation.

## Figure and ground

- The user must know what is content, what is background, and what is
  actionable.
- Borders, shadows, and tinted surfaces should clarify layers, not compensate
  for missing hierarchy.
- Empty space is a structural tool when it explains grouping and flow.

## Similarity

- Similar-looking objects will be interpreted as similar in function.
- Do not make a destructive action look like a neutral secondary action.
- Do not make low-priority metrics look identical to operational blockers.

## Continuity

- The scan path should follow the user's decision flow.
- Put summary before detail, decision before configuration, recovery near error,
  and action near the object being acted on.
- Responsive stacking must preserve priority, not merely preserve source order.

## Closure

- Containers, whitespace, and rhythm should help the user infer structure
  without extra borders around every object.
- Use containers for real grouping or state, not as a default decoration.

## Economy

- Remove visual decisions that do not clarify task, structure, state, or brand.
- Prefer one clear layout idea over multiple competing effects.
- An interface feels premium when it is specific, restrained, and resilient
  under real content and states.
